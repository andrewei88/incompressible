Autoresearch: Autonomous AI Research

github.com/karpathy · 1223 → 350 words · 2 min · 4 min saved
ARGUMENT · Key Points

Vision

  • Karpathy's framing: research was once done by “meat computers” synchronizing via “sound wave interconnect” in “group meeting.” That era is gone. Research is now “the domain of autonomous swarms of AI agents running across compute cluster megastructures.” “This repo is the story of how it all began.”
  • The idea: give an AI agent a small but real LLM training setup, let it experiment autonomously overnight. It modifies code, trains 5 minutes, checks if the result improved, keeps or discards, repeats. You wake up to a log of experiments and hopefully a better model.
  • You don't touch the Python files. You program program.md Markdown files that provide context to AI agents. The agent edits train.py autonomously.

Design Choices

ChoiceWhyTradeoff
Single file to modifyKeeps scope manageable and diffs reviewable
Fixed 5-min time budgetExperiments directly comparable regardless of what agent changes. Finds optimal model for your platform. ~12 experiments/hour, ~100 overnightResults not comparable across different compute platforms

SEQUENCE · Steps

Quick Start

  1. Requirements: single NVIDIA GPU (tested on H100), Python 3.10+, uv
  2. uv sync (install dependencies)
  3. uv run prepare.py (download data + train tokenizer, one-time, ~2 min)
  4. uv run train.py (single training run, ~5 min)
PROCESS · Key Points

Running the Agent

  • Spin up Claude/Codex (or whatever you want) in this repo, disable all permissions, prompt it to read program.md.
  • program.md is “a super lightweight skill” providing context and setting up the autonomous research org.
  • The default program.md is kept as a “bare bones baseline” though one would iterate on it to find the research org code that achieves the fastest progress.
PROCESS · Checklist

Tuning for Smaller Hardware

Recommendations for non-H100 platforms
☐ Use TinyStories dataset (lower entropy, GPT-4 generated short stories, reasonable results with smaller models)
☐ Decrease vocab_size: 8192 down to 4096, 2048, 1024, or 256 (byte-level tokenizer)
☐ Lower MAX_SEQ_LEN (down to 256); increase DEVICE_BATCH_SIZE to compensate (tokens per pass = product of both)
☐ Decrease EVAL_TOKENS so validation evaluates on less data
☐ Lower DEPTH (default 8, try 4). Many variables are functions of DEPTH
☐ Use WINDOW_PATTERN “L” only (“SSSL” alternating banded attention may be very inefficient)
☐ Lower TOTAL_BATCH_SIZE (powers of 2, down to 2^14 / ~16K)
